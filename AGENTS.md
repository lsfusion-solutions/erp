# AGENTS.md

Памятка для AI-агентов и инженеров, работающих в этом репозитории.

## Сопутствующие подпроекты

Если рядом с `erp` присутствует подпроект `custom-logics` (то есть файл
`../custom-logics/AGENTS.md` доступен на чтение) — прочитать его и применять

## Redmine (support.luxsoft.by)

База: `http://support.luxsoft.by` — внутренний Redmine, протокол **HTTP** (не HTTPS).

API-ключ берётся из пользовательской переменной окружения `REDMINE_API_KEY`.
Передавать через заголовок `X-Redmine-API-Key`, а не через query-параметр `key=...`
(чтобы не уходил в логи прокси/историю).

### Чтение задачи

```bash
curl -sS \
  -H "X-Redmine-API-Key: $REDMINE_API_KEY" \
  "http://support.luxsoft.by/issues/<ID>.json?include=journals,attachments,relations,children"
```

`include`-параметры:

| Значение | Что добавляет |
|---|---|
| `journals` | комментарии и историю изменений атрибутов |
| `attachments` | список вложений (с `content_url` для скачивания) |
| `relations` | связанные задачи (blocked/blocks/duplicates/...) |
| `children` | подзадачи (с их `subject` и `tracker`) |

### Полезные поля ответа

- `issue.subject`, `issue.description` — заголовок и описание.
- `issue.project.name`, `issue.tracker.name`, `issue.status.name`, `issue.priority.name`.
- `issue.author.name`, `issue.assigned_to.name` (может отсутствовать).
- `issue.parent.id` — родительская задача (читать рекурсивно при необходимости).
- `issue.journals[].notes` — текст комментария; `issue.journals[].details[]` — изменения атрибутов (description, status_id, assigned_to_id и т.п.).
- `issue.custom_fields[]` — пользовательские поля проекта.

### Поиск и фильтры

```bash
# Все задачи проекта в статусе "Новая"
curl -sS -H "X-Redmine-API-Key: $REDMINE_API_KEY" \
  "http://support.luxsoft.by/issues.json?project_id=<slug>&status_id=1&limit=100"
```

Пагинация: `limit` (макс. 100) + `offset`.

### Скачать вложение

```bash
curl -sS -H "X-Redmine-API-Key: $REDMINE_API_KEY" \
  "http://support.luxsoft.by/attachments/download/<attachment_id>/<filename>" \
  -o <filename>
```

### Завершение задачи: комментарий + перевод в «Решена»

После того как изменения закоммичены и в задачу добавлен комментарий с
кратким описанием для клиента (без технических подробностей; ревизию
указывать не нужно), **сразу же предложить пользователю** также
перевести задачу в статус **«Решена»** (`status_id=3`).

Принятая в команде конвенция: разработчик переводит в `Решена` сразу
после готовности изменений (изменения в svn + комментарий); клиент далее сам закрывает задачу или возвращает в работу.

Перевод статуса:

```bash
curl -sS -X PUT \
  -H "X-Redmine-API-Key: $REDMINE_API_KEY" \
  -H "Content-Type: application/json" \
  -d '{"issue":{"status_id":3}}' \
  "http://support.luxsoft.by/issues/<ID>.json"
```

Полный список статусов (`id` ↔ `name`) — `GET /issue_statuses.json`.

### Ограничения

- `WebFetch` на этот хост не работает (внутренний — `ECONNREFUSED` извне корпсети).
  Использовать `curl` через Bash с `$REDMINE_API_KEY`.
- API-ключ не логировать, не печатать в чат, не сохранять в файлы репозитория.

## Конвенции репозитория

- Источник правды для lsFusion-кода: `*/src/main/lsfusion`. Файлы в `out/production` —

### Обязательный вызов `lsfusion_get_guidance` перед работой с `.lsf`

Перед **любой** операцией с `.lsf`-файлом (Read, Edit, Write — то есть и чтение, и
анализ, и правка) агент **обязан** один раз за сессию вызвать
`mcp__lsfusion__lsfusion_get_guidance` и применять полученные правила

Это **жёсткое требование**, а не рекомендация. Причины:

- Тонкости lsFusion (трёхзначная логика, семантика сессий, конвенции именования)
  не выводятся из чтения кода — без guidance агент почти гарантированно даёт
  некорректные правки.
- «Просто посмотреть/процитировать» в lsFusion часто заканчивается неверным
  выводом о поведении кода — guidance нужен и для чтения.

#### Как соблюдать

1. Если `mcp__lsfusion__lsfusion_get_guidance` помечен как deferred — сначала
   подгрузи схему через `ToolSearch` запросом
   `select:mcp__lsfusion__lsfusion_get_guidance`.
2. Вызови инструмент **до** первого `Read`/`Edit`/`Write` по `.lsf`-файлу
   в текущей сессии.
3. Дальше в этой же сессии повторный вызов не требуется.

### Проверка результата правок `.lsf`: только запуск службы

В lsFusion успешная компиляция модуля не гарантирует корректность кода: значительная
часть ошибок (метасинтез, конфликты ABSTRACT/OVERRIDE, формы, политики безопасности,
миграции, события) проявляется только при старте сервера и инициализации логики.
Поэтому **критерий «задача решена» для .lsf — это успешный запуск службы**, а не
успех компиляции.

После завершения правок `.lsf` агент обязан:

1. Вызвать `mcp__sse__get_run_configurations` и показать пользователю доступные
   конфигурации (если инструмент помечен как deferred — сначала подгрузить схему
   через `ToolSearch` запросом
   `select:mcp__sse__get_run_configurations,mcp__sse__execute_run_configuration`).
2. **Предложить** пользователю запустить подходящую конфигурацию (как правило —
   серверную для текущего модуля). Имя конфигурации не угадывать: брать из ответа
   `get_run_configurations`. Если подходящих несколько — перечислить и спросить.
3. **Не запускать** конфигурацию самостоятельно. `mcp__sse__execute_run_configuration`
   вызывается только после явного согласия пользователя в этом же сообщении —
   старые согласия не переносятся.
4. Финальный отчёт о задаче формулировать так, чтобы было видно: правки внесены,
   но окончательная проверка — за запуском службы (а не за результатом сборки).

#### Как интерпретировать результат `execute_run_configuration`

`exitCode` и `success` от инструмента **ненадёжны** для lsFusion-сервера: при ошибке
инициализации логик сервер корректно завершает свой процесс (логирует ошибку,
останавливается), и инструмент возвращает `exitCode: 0`, `success: true`. Если
ориентироваться на эти поля — ошибка старта будет пропущена.

Достоверный сигнал — **только содержимое `output` (лога StartLogger)**:

- **Сбой инициализации** — в логе строка вида
  `ERROR StartLogger - Exception while starting logics instance: ...`
  и далее `Error starting server, server will be stopped`. Текст после
  `[error]:` — это собственно диагностика (имя модуля, позиция, причина).
  Типичные классы ошибок здесь: синтаксис (`missing EOF at '...'`,
  `mismatched input`), метасинтез, конфликты ABSTRACT/OVERRIDE, формы.
- **Успешный старт** — в логе нет `ERROR StartLogger` и присутствует сообщение
  о готовности сервера (`Server has successfully started` / эквивалент о слушающем порту).
  Если инструмент при этом отдал таймаут — это **нормально**: сервер не
  завершается, и таймаут означает «не упало быстро», а не «всё хорошо» —
  всё равно нужно убедиться по логу.
- **Лог обрывается до сообщения о готовности и без `ERROR StartLogger`** —
  трактовать как «непонятно»: либо ещё инициализируется, либо завис.
  Не отчитываться об успехе; либо увеличить `timeout`, либо попросить
  пользователя посмотреть Run-окно IDE.

Соответственно, при докладе пользователю **не цитировать `exitCode`/`success`
как доказательство успеха** — приводить выдержку из лога (наличие/отсутствие
`ERROR StartLogger` и сообщения о готовности).
