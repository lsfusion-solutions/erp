# AGENTS.md

Памятка для AI-агентов и инженеров, работающих в этом репозитории.

## Сопутствующие подпроекты

Если рядом с `erp` присутствует подпроект `custom-logics` (то есть файл
`../custom-logics/AGENTS.md` доступен на чтение) — прочитать его и применять
описанные там правила **в дополнение** к этому файлу. Если такого подпроекта
нет (стандартный случай открытой `erp`) — пропустить, всё необходимое уже
здесь.

## Redmine (support.luxsoft.by)

База: `http://support.luxsoft.by` — внутренний Redmine, протокол **HTTP** (не HTTPS).

API-ключ берётся из пользовательской переменной окружения `REDMINE_API_KEY`.
Передавать через заголовок `X-Redmine-API-Key`, а не через query-параметр `key=...`
(чтобы не уходил в логи прокси/историю).

### Чтение задачи

```bash
curl -sS \
  -H "X-Redmine-API-Key: $REDMINE_API_KEY" \
  "http://support.luxsoft.by/issues/<ID>.json?include=journals,attachments,relations,children"
```

`include`-параметры:

| Значение | Что добавляет |
|---|---|
| `journals` | комментарии и историю изменений атрибутов |
| `attachments` | список вложений (с `content_url` для скачивания) |
| `relations` | связанные задачи (blocked/blocks/duplicates/...) |
| `children` | подзадачи (с их `subject` и `tracker`) |

### Полезные поля ответа

- `issue.subject`, `issue.description` — заголовок и описание.
- `issue.project.name`, `issue.tracker.name`, `issue.status.name`, `issue.priority.name`.
- `issue.author.name`, `issue.assigned_to.name` (может отсутствовать).
- `issue.parent.id` — родительская задача (читать рекурсивно при необходимости).
- `issue.journals[].notes` — текст комментария; `issue.journals[].details[]` — изменения атрибутов (description, status_id, assigned_to_id и т.п.).
- `issue.custom_fields[]` — пользовательские поля проекта.

### Поиск и фильтры

```bash
# Все задачи проекта в статусе "Новая"
curl -sS -H "X-Redmine-API-Key: $REDMINE_API_KEY" \
  "http://support.luxsoft.by/issues.json?project_id=<slug>&status_id=1&limit=100"
```

Пагинация: `limit` (макс. 100) + `offset`.

### Скачать вложение

```bash
curl -sS -H "X-Redmine-API-Key: $REDMINE_API_KEY" \
  "http://support.luxsoft.by/attachments/download/<attachment_id>/<filename>" \
  -o <filename>
```

### Завершение задачи: комментарий + перевод в «Решена»

После того как изменения закоммичены и в задачу добавлен комментарий с
кратким описанием для клиента (без технических подробностей — в том
числе идентификаторов и других имён элементов; ревизию указывать тоже
не нужно; только видимые пользователю заголовки), **сразу же предложить
пользователю** также перевести задачу в статус **«Решена»**
(`status_id=3`).

Принятая в команде конвенция: разработчик переводит в `Решена` сразу
после готовности изменений (изменения в svn + комментарий); клиент далее сам закрывает задачу или возвращает в работу.

Перевод статуса:

```bash
curl -sS -X PUT \
  -H "X-Redmine-API-Key: $REDMINE_API_KEY" \
  -H "Content-Type: application/json" \
  -d '{"issue":{"status_id":3}}' \
  "http://support.luxsoft.by/issues/<ID>.json"
```

Полный список статусов (`id` ↔ `name`) — `GET /issue_statuses.json`.

### Ограничения

- `WebFetch` на этот хост не работает (внутренний — `ECONNREFUSED` извне корпсети).
  Использовать `curl` через Bash с `$REDMINE_API_KEY`.
- API-ключ не логировать, не печатать в чат, не сохранять в файлы репозитория.

## Конвенции репозитория

- Источник правды для lsFusion-кода: `*/src/main/lsfusion`. Файлы в `out/production` —
  генерируемые, править нельзя.

### Обязательный вызов `lsfusion_get_guidance` перед работой с `.lsf`

Перед **любой** операцией с `.lsf`-файлом (чтением, анализом, правкой) агент
**обязан** один раз за сессию вызвать `mcp__lsfusion__lsfusion_get_guidance` и
применять полученные правила.

Это **жёсткое требование**, а не рекомендация. Причины:

- Тонкости lsFusion (трёхзначная логика, семантика сессий, конвенции именования)
  не выводятся из чтения кода — без guidance агент почти гарантированно даёт
  некорректные правки.
- «Просто посмотреть/процитировать» в lsFusion часто заканчивается неверным
  выводом о поведении кода — guidance нужен и для чтения.

#### Как соблюдать

1. Вызвать `mcp__lsfusion__lsfusion_get_guidance` **до** первой операции с
   `.lsf`-файлом в текущей сессии.
2. Дальше в этой же сессии повторный вызов не требуется.

### Проверка результата правок `.lsf`: только запуск службы

В lsFusion успешная компиляция модуля не гарантирует корректность кода: значительная
часть ошибок (метасинтез, конфликты ABSTRACT/OVERRIDE, формы, политики безопасности,
миграции, события) проявляется только при старте сервера и инициализации логики.
Поэтому **критерий «задача решена» для .lsf — это успешный запуск службы**, а не
успех компиляции.

#### Процедура

1. Вызвать `mcp__sse__get_run_configurations`, показать пользователю доступные
   конфигурации.
2. **Предложить** подходящую конфигурацию (как правило — серверную для текущего
   модуля). Имя — из ответа `get_run_configurations`, не угадывать. Если подходящих
   несколько — спросить.
3. **Не запускать** конфигурацию самостоятельно. Запуск идёт только после явного
   согласия пользователя в этом же сообщении — старые согласия не переносятся.
4. Финальный отчёт оформлять так, чтобы было видно: правки внесены, окончательная
   проверка — за запуском службы (а не за результатом сборки).

#### Доступные механизмы запуска и остановки

- **`mcp__sse__xdebug_start_debugger_session(configurationName=…)`** — основной
  запуск. Подходит и для верификации без отладки. Возвращает `sessionId`,
  `state`, `fullOutputPath`. Главное преимущество — IDE-native канал статуса
  `mcp__sse__xdebug_get_debugger_status` (мгновенный ответ «жива ли сессия»).
  `state: running` сразу после старта означает только «сессия создана, JVM
  поднялась» — это **не** доказательство успешного старта lsFusion-логики,
  вердикт всё равно по логу.
- **`mcp__sse__execute_run_configuration(configurationName=…)`** — альтернатива
  без живого канала статуса. Возвращает `fullOutputPath`; единственный источник
  вердикта — лог.
- **Остановка своего запуска** — `mcp__sse__xdebug_control_session(action="STOP")`,
  если стартовали через `xdebug_*`. Сервер, поднятый из IDE до начала нашей сессии
  («orphan»), в `xdebug_get_debugger_status` не виден — гасить только средствами
  ОС, убивая процесс, держащий порт сервера (по умолчанию 7651); этот фолбэк
  **не использовать** для своих запусков.

#### Доступные источники лога

Два эквивалентных по содержимому:

- **`<MODULE_DIR>/logs/start.log`** — пишет сама lsFusion (StartLogger). Стабильный
  путь, не зависит от способа запуска. **Накопительный** (append-only): все запуски
  модуля подряд за всю историю. `MODULE_DIR` — каталог Maven-модуля; в сборках, где
  в одном Maven-модуле живут несколько top-модулей (общий `start.log`), маппинг
  «конфигурация → модуль» описан в `AGENTS.md` подпроекта (например,
  `custom-logics/AGENTS.md`).
- **`fullOutputPath`** из ответа стартового тулза — IDE-temp за конкретный запуск,
  имя уникальное.

В обоих источниках сообщения IDE-Run-окна вида `Process finished with exit code -1`
**не пишутся** — это только UI Run-окна.

#### Маркеры StartLogger

- Начало запуска: `INFO StartLogger - Server is starting...`
- Успех: `INFO StartLogger - Server has successfully started in N ms`
- Сбой инициализации: `ERROR StartLogger - Exception while starting logics instance: [error]:`,
  далее `Error starting server, server will be stopped`. Диагностика — на
  **следующих** строках после `[error]:` (имя модуля, позиция `Module:line:col`,
  причина). Типичные классы: синтаксис (`missing EOF at '...'`, `mismatched input`),
  метасинтез, конфликты `ABSTRACT`/`OVERRIDE`, формы, неизвестные классы
  (`class 'X' is not found`). Брать с контекстом следующих ~30 строк после
  строки `ERROR StartLogger`.

#### Три исхода

Поскольку `start.log` накопительный, при поиске маркеров **обязателен анкер
текущего запуска** — позиция в файле до старта, timestamp-фильтр, счётчик
матчей или иной механизм; конкретный способ здесь намеренно не предписан
(см. ниже).

- **Успех** — после анкера есть `Server has successfully started in N ms`. При
  запуске через `xdebug_*` дополнительно подтверждается
  `xdebug_get_debugger_status` → `state: running`.
- **Сбой инициализации** — после анкера есть `ERROR StartLogger ...`. При
  запуске через `xdebug_*` сессия в `xdebug_get_debugger_status` исчезает.
- **Внешнее прекращение** (Stop из IDE, OS kill, OOM) — после анкера нет ни
  успеха, ни ERROR; лог обрывается на стадии инициализации (`Initializing tables`,
  `Synchronizing tables started`, …). При запуске через `xdebug_*` дополнительный
  признак — пустой `xdebug_get_debugger_status`. Распознаётся **только** по
  сочетанию «маркеров в логе нет + (если стартовали через `xdebug_*`) сессии в
  `xdebug_get_debugger_status` нет».

Если маркеров после анкера нет, а сессия в `xdebug_get_debugger_status` всё ещё
`state: running` — сервер ещё инициализируется (типичный случай — долгая
миграция / синхронизация таблиц после изменения `modulesHash`, минуты).
Не отчитываться об успехе.

#### Что НЕ цитировать как доказательство успеха

`exitCode`, `success`, `state` от стартового тулза — **ненадёжны** для
lsFusion-сервера: при ошибке инициализации сервер корректно завершает процесс,
и для тулза это выглядит как нормальный выход. Доказательство — только
анкерованные маркеры в логе + при запуске через `xdebug_*` — статус сессии в
`xdebug_get_debugger_status`.

### Свёрнутый метакод в коммитах `.lsf`

В коммит должны попадать только **свёрнутые** обращения к метакоду — одна строка
вида `@metaName(arg1, arg2, ...);`. Развёрнутая форма (когда плагин lsFusion в
IDEA «расклеил» тело метакода в исходник) в репозиторий идти не должна: она
дублирует сгенерированный код, ломает дальнейшие правки шаблона и засоряет diff.

**Признак раскрытого метакода — фигурная скобка `{` сразу после `@metaName(args)`**
(вместо `;`). По синтаксису lsFusion корректный вызов метакода — это всегда
`@metaName(args);` без блока; форма `@metaName(args){ … }` появляется только
когда плагин IDEA вставил тело метакода в исходник.

При обнаружении `@name(args){` — **не коммитить**. Сообщить пользователю с
указанием файла и попросить выполнить в IDEA `Hide Meta for File` (или
`Reenable meta`, если плагин не схлопнул автоматически). Прямого MCP-инструмента
для сворачивания метакода нет — действие доступно только из плагина lsFusion
в IDEA.

Никогда не разворачивать метакод самостоятельно ради правок «пары строк внутри».
Если нужно изменить то, что генерируется, — менять сам `META`-шаблон, а не его
раскрытый результат.
